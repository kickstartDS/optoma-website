# PRD: Unified W3C Theming Architecture

## Problem Statement

The theming system spans five systems (Design System, Design Tokens MCP, Storyblok MCP, Design Tokens Editor, website runtime) but uses **two incompatible token formats** and has **no unified flow** for creating, storing, and applying themes. The Design Tokens MCP operates on a local flat-format file (`branding-token.json`) that is disconnected from the W3C Design Token JSON stored in Storyblok `token-theme` stories. There is also no concept of a code-managed "default" theme — the Design System's baseline tokens exist only as baked-in CSS, invisible to the Editor and MCP tooling.

## Goals

1. **W3C Design Token JSON everywhere** — All systems speak the same token format, eliminating conversion gaps.
2. **Ephemeral Design Tokens MCP** — The Design Tokens MCP becomes stateless: it analyzes, transforms, and validates tokens but does not own storage. Themes live in Storyblok.
3. **Storyblok MCP gains theme CRUD** — `create_theme` and `update_theme` tools enable the LLM to persist themes generated by the Design Tokens MCP.
4. **Code-managed default theme** — The Design System defines the canonical default theme in code. It is synced to Storyblok on seeding and every build, and is protected from overwrite in the Editor and MCP.

## Non-Goals

- Changing the website runtime's CSS injection mechanism (the `<style data-tokens>` approach stays).
- Modifying component-level tokens (`--dsa-*`) — only branding tokens are in scope.
- Changing how Storyblok stores `token-theme` stories (same content type, same `tokens` + `css` fields).
- Replacing the Design Tokens Editor UI — it already speaks W3C natively.

## Current State

### Format Landscape

| System                   | Format                                     | Storage                                                    | Can Create Themes?                  | Can Apply Themes?                   |
| ------------------------ | ------------------------------------------ | ---------------------------------------------------------- | ----------------------------------- | ----------------------------------- |
| **Design System**        | W3C Design Token JSON                      | `src/token/branding-tokens.json` (source), compiled to CSS | N/A (source of truth)               | N/A                                 |
| **Design Tokens MCP**    | Flat nested JSON (`color.primary: "#hex"`) | Local `branding-token.json` file                           | Writes to local file only           | No                                  |
| **Design Tokens Editor** | W3C Design Token JSON                      | Storyblok `token-theme` stories                            | Yes (full CRUD via Express backend) | No                                  |
| **Storyblok MCP**        | Reads W3C from `token-theme` stories       | Storyblok CMS                                              | No (read + apply only)              | Yes (`apply_theme`, `remove_theme`) |
| **Website runtime**      | CSS custom properties                      | Baked into `global.css` + `<style data-tokens>`            | N/A                                 | Consumes themes                     |

### How Theme Generation Works Today

Both `generate_theme_from_image` and `extract_theme_from_css` in the Design Tokens MCP are **data-gathering helpers**, not generators. They return raw data (image + schema guidance, or parsed CSS + schema guidance) to the LLM. The LLM decides on values, then calls `update_theme_config(path, value)` once per field to write to the local `branding-token.json`. This result never reaches Storyblok.

### The Two-Default Problem

Two separate "defaults" exist today:

- **Design System**: `packages/design-system/src/token/branding-tokens.json` — the canonical token set shipped with the Design System
- **Website**: `packages/website/token/branding-token.json` — a per-project customization compiled into `global.css` at build time

Neither is represented as a `token-theme` story in Storyblok. The Design Tokens Editor and MCP tools have no way to see or reference the baseline.

**Resolution:** The website-specific `branding-token.json` is a historical artifact from when the Design System could serve multiple independent website projects. Now that this is a monorepo serving exactly one project, the website-specific file will be **removed**. The Design System's `branding-tokens.json` becomes the single source of truth for the default theme.

## Proposed Architecture

### Part 1: W3C Format Everywhere

#### W3C Design Token JSON Format (Reference)

All systems will use this structure:

```json
{
  "color": {
    "primary": {
      "$root": {
        "$type": "color",
        "$value": { "colorSpace": "srgb", "components": [0.19, 0.4, 0.75] }
      },
      "inverted": {
        "$type": "color",
        "$value": { "colorSpace": "srgb", "components": [0.19, 0.4, 0.75] }
      }
    },
    "background": {
      "$root": {
        "$type": "color",
        "$value": { "colorSpace": "srgb", "components": [1, 1, 1] }
      },
      "inverted": {
        "$type": "color",
        "$value": { "colorSpace": "srgb", "components": [0.06, 0.13, 0.24] }
      }
    }
  },
  "font": {
    "family": {
      "display": {
        "$type": "fontFamily",
        "$value": ["Montserrat", "sans-serif"]
      },
      "copy": {
        "$type": "fontFamily",
        "$value": ["system-ui", "-apple-system", "sans-serif"]
      }
    },
    "weight": {
      "regular": { "$type": "fontWeight", "$value": 400 },
      "bold": { "$type": "fontWeight", "$value": 700 }
    },
    "size": {
      "display": {
        "base": {
          "$type": "dimension",
          "$value": { "value": 20, "unit": "px" }
        }
      }
    }
  },
  "spacing": {
    "base": { "$type": "dimension", "$value": { "value": 14, "unit": "px" } },
    "scale-ratio": { "$type": "number", "$value": 1.375 }
  }
}
```

Validated against the existing JSON Schema at `packages/design-system/src/token/branding-tokens.schema.json`.

#### 1.1 Design Tokens MCP → Stateless W3C

**Current:** Reads/writes a local `branding-token.json` in flat format. Stateful — stores theme in a file.

**After:** All tools accept and return W3C token objects. No local file. The MCP is a pure analysis + transformation service.

##### Tool Changes

| Tool                        | Current Behavior                                                 | New Behavior                                                                                                                |
| --------------------------- | ---------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| `generate_theme_from_image` | Returns image + flat schema guidance, references local file      | Returns image + **W3C schema guidance** (paths, types, value ranges from `branding-tokens.schema.json`). No file reference. |
| `extract_theme_from_css`    | Scrapes CSS, returns flat schema guidance, references local file | Scrapes CSS, returns **W3C schema guidance**. No file reference.                                                            |
| `update_theme_config`       | Reads flat JSON, sets value at dot-path, writes to disk          | **Removed.** The LLM constructs the full W3C object in-context; no per-field roundtrips needed.                             |
| `get_theme_config`          | Returns contents of local file                                   | **Replaced by `get_theme_schema`**: returns the W3C schema structure with field descriptions and valid ranges. No file I/O. |
| `list_theme_values`         | Flattens local file to path/value pairs                          | Accepts a W3C token object as input, returns a flattened summary. Stateless.                                                |
| `validate_theme`            | _Does not exist_                                                 | **New.** Validates a W3C token object against `branding-tokens.schema.json`. Returns pass/fail + field-level errors.        |

##### File Changes

| File                         | Change                                                                                                                                                  |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `branding.ts`                | Remove `readBrandingJson()` / `writeBrandingJson()` / `setNestedValue()`. Add `flattenW3CTokens()`, `getW3CSchemaDescription()`, `validateW3CTokens()`. |
| `handlers.ts`                | Update 6 tool handlers to work with W3C objects. Remove file I/O.                                                                                       |
| `resources.ts`               | `tokens://branding` resource returns the W3C schema (reference structure) instead of a local file's contents.                                           |
| `prompts.ts`                 | `update-branding` prompt updated to describe W3C format and reference `get_theme` (Storyblok MCP) + `create_theme` (Storyblok MCP).                     |
| `tokens/branding-token.json` | **Removed.** No longer needed — the MCP is stateless.                                                                                                   |

##### New Workflow Pattern

```
LLM calls Design Tokens MCP: generate_theme_from_image(imageUrl)
  → Returns: image + W3C schema guidance (field paths, types, value constraints)

LLM analyzes image, constructs complete W3C token object in-context

LLM calls Design Tokens MCP: validate_theme(tokens)
  → Returns: validation result

LLM calls Storyblok MCP: create_theme(name: "brand-blue", tokens: { W3C object })
  → Storyblok MCP: tokensToCss(tokens) → CSS, stores both in Storyblok

LLM calls Storyblok MCP: apply_theme(storyId, themeSlug: "brand-blue")
```

**Key improvement:** The LLM builds the entire token object at once and persists it in a single call — no more 15+ `update_theme_config` roundtrips writing to a local file that goes nowhere.

#### 1.2 Storyblok MCP → Theme CRUD

**Current:** `list_themes`, `get_theme`, `apply_theme`, `remove_theme` (read + apply only).

**After:** Add `create_theme` and `update_theme`.

##### New Tools

**`create_theme`**

| Property   | Value                                                                                                                                                                                                             |
| ---------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Input      | `name: string`, `tokens: object` (W3C Design Token JSON)                                                                                                                                                          |
| Process    | 1. Call `tokensToCss(tokens)` to compile CSS. 2. Create `token-theme` story at `settings/themes/{slug}` with `tokens` (serialized) + `css` fields. 3. Auto-create `settings/themes/` folder hierarchy if missing. |
| Output     | `{ success, storyId, slug, fullSlug }`                                                                                                                                                                            |
| Protection | Reject if `name` resolves to a protected slug (see Part 2).                                                                                                                                                       |

**`update_theme`**

| Property | Value                                                                                                                                         |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| Input    | `slugOrUuid: string`, `tokens: object` (W3C Design Token JSON)                                                                                |
| Process  | 1. Resolve theme story. 2. Reject if protected (see Part 2). 3. Recompile CSS via `tokensToCss()`. 4. Update story with new `tokens` + `css`. |
| Output   | `{ success, storyId, slug, previousTokens (summary), newTokens (summary) }`                                                                   |

##### Implementation Location

The core `createTheme()` and `updateTheme()` functions go into **`packages/storyblok-services/src/themes.ts`** (where `listThemes`, `getTheme`, `applyTheme` already live). This gives both the Storyblok MCP and the Design Tokens Editor the same shared implementation.

**Dependency:** `tokensToCss()` is imported from `@kickstartds/design-system/scripts/tokensToCss.mjs` — already a workspace dependency of `storyblok-mcp`.

##### Design Tokens Editor Alignment

The Editor's standalone CRUD in `packages/design-tokens-editor/src/server/storyblok.ts` can be refactored to delegate to the shared `storyblok-services` functions, eliminating duplicated Storyblok API calls and folder-creation logic.

#### 1.3 n8n Node → Theme CRUD

The n8n node's Theme resource (currently: list, get, apply, remove) gains two operations:

- `create` — Calls shared `createTheme()` from `storyblok-services`
- `update` — Calls shared `updateTheme()` from `storyblok-services`

Same validation, same `tokensToCss()` compilation, same protection checks.

### Part 2: Code-Managed Default Theme

#### Concept

The Design System defines a canonical default theme as `branding-tokens.json`. This is synced to a `token-theme` story in Storyblok (`settings/themes/default`) at two points:

1. **Project seeding** (`prepareProject.js` or equivalent) — initial creation
2. **Every website build** — keeps Storyblok in sync with code changes

The default theme story is **protected** — it cannot be overwritten or deleted through the Design Tokens Editor or MCP tools. Editors can load it and "Save As" to create a derivative.

#### Source of Truth

> **Decided: Option A — Design System's `branding-tokens.json`.**
>
> The website-specific `packages/website/token/branding-token.json` is a historical artifact from when the Design System could serve multiple independent website projects. Now that this is a monorepo serving exactly one project, maintaining a separate website-level copy is unnecessary duplication. The website-specific file will be **removed**, and the website's build pipeline (`build-tokens`, `global.css` compilation) will read directly from the Design System's `packages/design-system/src/token/branding-tokens.json`. Per-project customization happens by creating and applying a separate theme in Storyblok.

#### Sync Script: `sync-default-theme`

A new idempotent script that:

1. Reads the Design System's `packages/design-system/src/token/branding-tokens.json`
2. Validates against `branding-tokens.schema.json`
3. Compiles CSS via `tokensToCss()`
4. Upserts the `settings/themes/default` story in Storyblok:
   - Creates if missing (including folder hierarchy)
   - Updates if content has changed (compares token JSON)
   - Skips if identical (avoids unnecessary publishes)
5. Sets a `system: true` field on the story content (see Protection below)

**Called from:**

- `prepareProject.js` (project seeding — first-time setup)
- Website `build` script (after `build-tokens`, before `next build`)

**Requires:** `NEXT_STORYBLOK_OAUTH_TOKEN` (Management API) — already available in the build environment.

**Location:** `packages/website/scripts/syncDefaultTheme.ts` — calls shared functions from `storyblok-services`.

#### Protection Mechanism

##### Content Type Extension

Add a `system` boolean field to the `token-theme` content type schema:

```json
{
  "properties": {
    "system": {
      "title": "System Theme",
      "description": "System-managed theme — cannot be modified through the Editor or MCP",
      "type": "boolean",
      "default": false
    }
  }
}
```

The `sync-default-theme` script sets `system: true` on the default theme story.

##### Guard Points

| System                            | Guard                                                                                                                     |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| **Storyblok MCP** `update_theme`  | Check `story.content.system === true` → reject with error message                                                         |
| **Storyblok MCP** `delete_story`  | Check if target is a `token-theme` with `system: true` → reject                                                           |
| **Design Tokens Editor** backend  | `PUT /api/tokens/:name` and `DELETE /api/tokens/:name` → check `system` field → return 403                                |
| **Design Tokens Editor** frontend | Show "default" with a lock icon. Disable "Save" and "Delete". "Save As" remains enabled (creates a fork with a new name). |
| **n8n Node**                      | `update` and `delete` theme operations → check `system` field → reject                                                    |

##### Why `system` field over slug convention?

A `system: true` field is more robust than reserving the slug "default" because:

- The slug convention is implicit — a new developer could accidentally delete it
- The field is explicit and survives renames
- Future system themes (e.g., "dark-mode-default") can also be marked `system: true`

#### Runtime Behavior

**Small runtime optimization for the default theme.** The CSS injection cascade becomes:

1. Baked-in CSS from `global.css` (compiled from the Design System's `branding-tokens.json` at build time) — always present
2. Global theme CSS from `settings.theme` → `<style data-tokens>` — overrides if set, **but skipped when the selected theme is `default`** (since its CSS is already baked into `global.css`)
3. Page-level theme CSS from `pageContent.theme` → `<style data-tokens>` — overrides if set

Pages with no theme selected get the baked-in CSS automatically.

> **Decided: Explicit reference, but skip redundant injection.**
>
> The `settings` story explicitly references `theme: "default"`. Editors see "Default" selected in the theme dropdown and can switch to any other theme. However, the website runtime **does not inject** the default theme's CSS via `<style data-tokens>` — it recognizes the `default` slug and skips injection, since the identical CSS is already baked into `global.css`. This avoids redundant CSS while keeping the theme chain fully visible in the CMS.
>
> **Follow-up consideration:** The same optimization could apply to page-level themes. If a page explicitly selects `default` as its theme, injection should also be skipped. More broadly, when a page selects a non-default theme, the baked-in `global.css` CSS is still loaded first and then overridden by the `<style data-tokens>` injection — this is by design, as the theme CSS only contains overrides, not a complete replacement. The cascade ensures all tokens have values even if the theme only customizes a subset.

## Enabled Workflows (After Implementation)

### Workflow A: Generate Theme from Image

```
1. LLM → Design Tokens MCP: generate_theme_from_image(imageUrl)
     ← image + W3C schema guidance

2. LLM: analyzes image, constructs W3C token object

3. LLM → Design Tokens MCP: validate_theme(tokens)
     ← pass / fail + errors

4. LLM → Storyblok MCP: create_theme(name: "brand-blue", tokens: {...})
     → tokensToCss() → Storyblok story created

5. LLM → Storyblok MCP: apply_theme(storyId, themeSlug: "brand-blue")
     → Page themed
```

### Workflow B: Generate Theme from Existing Website

```
1. LLM → Design Tokens MCP: extract_theme_from_css(url: "https://example.com")
     ← parsed CSS (colors, fonts, custom properties) + W3C schema guidance

2. LLM: maps extracted values → W3C token object

3. LLM → Design Tokens MCP: validate_theme(tokens)
     ← validation result

4. LLM → Storyblok MCP: create_theme(name: "competitor-style", tokens: {...})
     → stored in Storyblok with pre-compiled CSS
```

### Workflow C: Edit Existing Theme

```
1. LLM → Storyblok MCP: get_theme(slugOrUuid: "brand-blue")
     ← { tokens: "{W3C JSON}", css: "..." }

2. LLM: parses tokens, makes modifications

3. LLM → Design Tokens MCP: validate_theme(tokens)
     ← confirms changes valid

4. LLM → Storyblok MCP: update_theme(slugOrUuid: "brand-blue", tokens: {...})
     → CSS recompiled, Storyblok updated
```

### Workflow D: Fork Default Theme

```
1. LLM → Storyblok MCP: get_theme("default")
     ← system theme tokens

2. LLM: modifies values (e.g., change primary color)

3. LLM → Storyblok MCP: create_theme(name: "custom-brand", tokens: {...})
     → new theme created (default untouched)

4. LLM → Storyblok MCP: apply_theme(storyId, themeSlug: "custom-brand")
```

### Workflow E: Theme Audit

```
1. LLM → Storyblok MCP: list_themes()
     ← all theme summaries (including "default" marked as system)

2. LLM → Storyblok MCP: get_theme("brand-blue")
     ← W3C tokens

3. LLM → Design Tokens MCP: list_theme_values(tokens)
     ← flattened summary

4. LLM → Design Tokens MCP: validate_theme(tokens) + audit_tokens()
     ← quality analysis (contrast ratios, consistency, etc.)
```

### Workflow F: Design Tokens Editor

```
1. Editor: loads theme list (sees "Default" with lock icon)
2. Editor: loads "Default" into workspace
3. Editor: makes visual modifications (color picker, font selector, etc.)
4. Editor: "Save As" → prompts for new name → POST /api/tokens/new-theme
     → storyblok-services: createTheme() → tokensToCss() → Storyblok
5. Editor: cannot "Save" or "Delete" the default (403 from backend)
```

### Workflow G: n8n Automation

```
1. n8n trigger (e.g., webhook, schedule)
2. n8n: Design Tokens MCP → extract_theme_from_css(competitor URL)
3. n8n: construct W3C token object
4. n8n: Storyblok node → create theme
5. n8n: Storyblok node → apply theme to staging pages
```

## Resolved Open Questions

### OQ1: Default Theme Source of Truth

**Decision: A — Design System's `branding-tokens.json`.**

The website-specific `packages/website/token/branding-token.json` is a historical artifact from when the Design System could serve multiple independent website projects. Now that this is a monorepo serving exactly one project, maintaining a separate copy creates unnecessary duplication and drift risk. The website-specific file will be **removed**, and the website's build pipeline will read directly from `packages/design-system/src/token/branding-tokens.json`. Per-project customization is achieved by creating a separate theme in Storyblok and applying it.

### OQ2: Explicit vs. Implicit Default in Settings

**Decision: Explicit reference, but skip redundant CSS injection.**

The `settings` story explicitly references `theme: "default"` so editors see it in the theme dropdown and can switch away from it. However, the website runtime **skips `<style data-tokens>` injection** when the resolved theme slug is `default`, since the identical CSS is already baked into `global.css` at build time. This avoids redundant CSS while keeping the theme chain visible in the CMS.

**Follow-up:** The same logic applies at the page level — if a page explicitly selects `default`, injection is also skipped. For non-default themes, the baked-in `global.css` provides the baseline and `<style data-tokens>` overrides specific tokens. This cascade is by design: theme CSS contains overrides, not complete replacements.

### OQ3: Default Theme Slug

**Decision: A — Always `default`.** Simple convention, universally understood.

### OQ4: Website's `branding-token.json` Format

**Decision: Not applicable.** Since the answer to OQ1 is A (Design System is the source of truth), the website-specific `branding-token.json` is being removed entirely. There is no website-level token file to migrate. The Design System's `branding-tokens.json` is already in W3C format.

### OQ5: Component Token Tools

**Decision: Stay as-is.** Component tokens remain as CSS custom properties. There is explicitly no intention of moving them to W3C format. They are a separate layer, unaffected by this migration.

### OQ6: Token Validation Strictness

**Decision: A — Strict.** Every required field must be present, all values must be within specified ranges. Partial themes are rejected. This applies uniformly to both `validate_theme` and `create_theme` — a theme must be complete to be valid.

### OQ7: Build-Time Storyblok Credential Requirement

**Decision: Skip with a warning.** If `NEXT_STORYBLOK_OAUTH_TOKEN` is not set, the `sync-default-theme` script logs a warning and exits successfully. The build should not fail due to missing Storyblok credentials (e.g., during local design system development or CI builds that don't need CMS access).

### OQ8: Migration of Existing Themes

**Decision: Absence = false. No migration needed.** Existing themes created by the Design Tokens Editor are already in W3C format. The `system` field won't exist on them, and its absence is treated as `false`. No backfill required.
